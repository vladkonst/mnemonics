## 1. Введение

### 1.1 Назначение документа

Данный документ представляет собой полную техническую спецификацию системы подготовки к экзаменам по анатомии. Документ описывает все пользовательские сценарии (user flows) с учётом всех выявленных требований и предложенных доработок.

### 1.2 Целевая аудитория

- Заказчик (для утверждения функциональности)
- Команда разработки (для реализации)

### 1.3 Глоссарий

| Термин | Определение |
|--------|-------------|
| **User Flow** | Последовательность действий пользователя для достижения цели |
| **Endpoint** | HTTP API endpoint для взаимодействия Telegram-бота с backend |
| **Мнемоника** | Учебный материал (текст или изображение) для запоминания |
| **Введение** | Специальная первая тема каждого модуля с терминами и мнемониками, обязательная для прохождения |
| **Тема** | Учебная единица, содержащая мнемоники и тест |
| **Модуль** | Группа связанных тем (первая тема - всегда "Введение") |
| **Промокод** | Код активации для университетского доступа |
| **Подписка** | Платный доступ ко всему контенту |

## 2. Общее описание системы

### 2.1 Архитектура

Система состоит из следующих компонентов:

```
┌─────────────┐      ┌──────────────┐      ┌─────────────────┐
│  Студент/   │ ───> │ Telegram Bot │ ───> │ Backend Service │
│ Преподаватель│      │   (Client)   │      │    (Go + DB)    │
└─────────────┘      └──────────────┘      └─────────────────┘
                            │                      │
                            │                      ├──> SQLite Database
                            │                      ├──> S3 Storage (изображения)
                            │                      └──> Payment Gateway (Stripe/YooKassa)
                            │
                            └──────────────────────────> Telegram API
```

### 2.2 Роли пользователей

| Роль | Описание | Возможности |
|------|----------|-------------|
| **Студент** | Изучает материалы, проходит тесты | Доступ к контенту, тестирование, прогресс, подписка |
| **Преподаватель** | Управляет группой студентов | Активация промокодов, просмотр прогресса студентов |
| **Администратор** | Управляет системой | Управление контентом, пользователями, подписками через GoAdmin |

**Примечание**: Пользователь может иметь обе роли (студент + преподаватель) одновременно.

### 2.3 Модель доступа к контенту

| Тип доступа | Способ получения | Ограничения |
|-------------|------------------|-------------|
| **Бесплатный** | По умолчанию | Последовательное открытие тем после прохождения тестов |
| **Университетский** | Промокод от преподавателя | Полный доступ на период действия промокода |
| **Персональная подписка** | Оплата через Telegram | Полный доступ на период подписки |

### 2.4 Структура учебного контента

Система организует контент в иерархическую структуру:

```
Модуль (например, "Остеология")
  └─ Введение (первая тема модуля) ⭐
       ├─ Термины (например: "диафиз", "эпифиз", "метафиз")
       ├─ Мнемоники для терминов
       └─ Тест на проверку знания терминов
  └─ Тема 2: "Позвоночник"
       ├─ Мнемоники
       └─ Тест
  └─ Тема 3: "Череп"
       └─ ...
```

#### Особенности "Введения":
- **Назначение**: Содержит ключевые термины с мнемониками для облегчения изучения последующих тем
- **Позиция**: Всегда первая тема в модуле (order_num = 1, is_introduction = true)
- **Прогресс**: Учитывается в общем прогрессе как обычная тема

#### Бизнес-логика доступа к темам:
- **Пользователи С подпиской**: Полный доступ ко всем темам модуля (включая введение) без ограничений
- **Пользователи БЕЗ подписки**: Последовательное открытие тем после прохождения тестов (начиная с введения как первой темы)

## 3. User Flow 1: Регистрация пользователя

### 3.1 Описание

Первое взаимодействие пользователя с системой: регистрация профиля и выбор роли.

### 3.2 Участники

- **Актор**: Новый пользователь
- **Система**: Telegram Bot, Backend Service, SQLite Database

### 3.3 Предусловия

- Пользователь нашёл бота в Telegram
- Бот активен и доступен

### 3.4 Основной сценарий

| Шаг | Действие пользователя | UI компонент | Backend endpoint | Параметры запроса | Изменяемые сущности | Возвращаемый результат |
|-----|----------------------|-------------|-----------------|-------------------|---------------------|----------------------|
| 1 | Пользователь открывает бота и нажимает "Start" | Команда `/start` | `POST /api/v1/users` | `telegram_id`, `first_name`, `last_name`, `username` | `users` (INSERT: telegram_id, role=student, subscription_status=inactive) | `{user_id, role, subscription_status, created_at}` (201 Created + Location header) |
| 2 | Система проверяет существование пользователя | - | Внутренняя логика: `SELECT FROM users WHERE telegram_id=?` | - | - | Пользователь существует / не существует |
| 3 | Бот показывает приветствие и выбор роли | Сообщение + Inline keyboard: "Я студент" / "Я преподаватель" | - | - | - | - |
| 4 | Пользователь выбирает роль | Callback: `set_role_student` или `set_role_teacher` | `PATCH /api/v1/users/{telegram_id}` | `role: "student"` или `role: "teacher"` | `users` (UPDATE: role, updated_at) | `{user_id, role, updated_at}` |
| 5 | Бот показывает главное меню | Inline keyboard с основными командами | - | - | - | Главное меню для выбранной роли |

### 3.5 Альтернативные сценарии

**3.5.1 Повторный запуск /start**

| Условие | Действие системы |
|---------|------------------|
| Пользователь уже зарегистрирован | Пропустить шаги 1-2, показать главное меню без выбора роли |

### 3.6 Зависимости

#### Backend Endpoints
- `POST /api/v1/users` (новый)
- `PATCH /api/v1/users/{telegram_id}` (новый)

#### База данных
- Таблица `users`: SELECT, INSERT, UPDATE
- Поля: `telegram_id` (PK), `role`, `subscription_status`, `created_at`, `updated_at`

#### Внешние сервисы
- Telegram Bot API: получение информации о пользователе

### 3.7 Диаграмма последовательности

```
Пользователь       Telegram Bot       Backend           Database
    │                   │                 │                 │
    ├──/start──────────>│                 │                 │
    │                   ├─POST /users────>│                 │
    │                   │                 ├──SELECT user──>│
    │                   │                 │<───not found────┤
    │                   │                 ├──INSERT user──>│
    │                   │<──201+Location──┤<───ok───────────┤
    │<──"Выберите роль"─┤                 │                 │
    ├──"Я студент"─────>│                 │                 │
    │                   ├─PATCH /users───>│                 │
    │                   │  /{id}          ├──UPDATE role──>│
    │                   │<───updated──────┤<───ok───────────┤
    │<──Главное меню────┤                 │                 │
```

### 3.8 Валидация и ошибки

| Ошибка | HTTP код | Описание | Действие системы |
|--------|----------|----------|------------------|
| Пользователь уже существует | 409 Conflict | telegram_id уже в БД | Показать главное меню |
| Невалидный telegram_id | 400 Bad Request | Некорректный формат ID | Показать сообщение об ошибке |
| Невалидная роль | 400 Bad Request | role не в enum(student, teacher) | Показать сообщение об ошибке |

### 3.9 Постусловия

- ✅ Пользователь зарегистрирован в системе
- ✅ Выбрана роль (student или teacher)
- ✅ Показано главное меню

## 4. User Flow 2: Изучение материала и прохождение теста

### 4.1 Описание

Основной учебный процесс: студент выбирает модуль, изучает тему (мнемоники), проходит тест и получает доступ к следующей теме.

### 4.2 Участники

- **Актор**: Студент
- **Системы**: Telegram Bot, Backend Service, SQLite Database, S3 Storage

### 4.3 Предусловия

- Пользователь зарегистрирован и вошёл в систему
- Существует хотя бы один модуль с темами
- Тема содержит мнемоники и тест

### 4.4 Основной сценарий

#### Этап 1: Выбор модуля и темы

| Шаг | Действие пользователя | UI компонент | Backend endpoint | Параметры | Изменяемые сущности | Возвращаемый результат |
|-----|----------------------|-------------|-----------------|-----------|---------------------|----------------------|
| 1.1 | Студент запрашивает список модулей | `/modules` | `GET /api/v1/content/modules?user_id={telegram_id}` | `user_id` | - | `[{module_id, name, description, total_themes, completed_themes}]` |
| 1.2 | Бот показывает модули с прогрессом | Inline keyboard: "Модуль 1 [30%]", "Модуль 2 [0%] 🔒" | - | - | - | - |
| 1.3 | Студент выбирает модуль | Callback: `module_select_{module_id}` | `GET /api/v1/content/modules/{module_id}/themes?user_id={telegram_id}` | `module_id`, `user_id` | - | `[{theme_id, name, order_num, is_introduction, is_completed, score}]` |
| 1.4 | Бот показывает список тем | Inline keyboard: "Введение (пройдено)", "Тема 2: Позвоночник (в процессе)", "Тема 3: Череп (недоступно)" | - | - | - | Введение помечено как первая обязательная тема |

#### Этап 2: Изучение мнемоник

| Шаг | Действие пользователя | UI компонент | Backend endpoint | Параметры | Изменяемые сущности | Возвращаемый результат |
|-----|----------------------|-------------|-----------------|-----------|---------------------|----------------------|
| 2.1 | Студент выбирает тему | Callback: `theme_view_{theme_id}` или `/study module=1 theme=2` | `POST /api/v1/users/{user_id}/study-sessions` | `theme_id` (в body) | `user_progress` (INSERT/UPDATE: status=started, started_at=NOW()) | `{session_id, theme: {id, name, description, is_introduction}, mnemonics: [{type, content_text, s3_image_url}], test_available: true, next_step: {type: "test", test_id}}` (201 Created) |
| 2.2 | Backend проверяет доступ | - | Внутренняя логика: <br>1) Проверка подписки/промокода<br>2) Для пользователей БЕЗ подписки: проверка последовательности тем (order_num) | - | - | access_granted / access_denied |
| 2.3 | Backend загружает мнемоники | - | SELECT FROM mnemonics + S3 pre-signed URLs | - | - | - |
| 2.4 | Бот отправляет мнемоники | Текст + изображения + кнопка "Начать тест" | - | - | - | - |

#### Этап 3: Прохождение теста

| Шаг | Действие пользователя | UI компонент | Backend endpoint | Параметры | Изменяемые сущности | Возвращаемый результат |
|-----|----------------------|-------------|-----------------|-----------|---------------------|----------------------|
| 3.1 | Студент нажимает "Начать тест" | Callback: `start_test_{theme_id}` или `/test_ready theme=2` | `POST /api/v1/users/{user_id}/test-attempts` | `theme_id` (в body) | `user_progress` (UPDATE: test_started_at=NOW(), current_attempt++) | `{attempt_id, test: {id, questions: [{id, question, options}], passing_score}, attempt_number, started_at}` (201 Created + Location header) |
| 3.2 | Бот показывает вопросы теста | Inline keyboard с вариантами ответов | - | - | - | - |
| 3.3 | Студент отвечает на вопросы | Последовательный выбор ответов | - | - | - | - |
| 3.4 | Студент отправляет ответы | Callback: `submit_test` или `/submit_answers` | `PUT /api/v1/users/{user_id}/test-attempts/{attempt_id}` | `answers: [{question_id, answer}]` (в body) | `user_progress` (UPDATE: score, status=completed/failed, completed_at), `test_attempts` (INSERT история попытки) | `{result: {score, passed, correct_answers, total_questions}, next_action: {type, theme_id, theme_name, message}, motivation_message}` (200 OK, idempotent) |

#### Этап 4: Разблокировка следующей темы

| Шаг | Действие пользователя | UI компонент | Backend endpoint | Параметры | Изменяемые сущности | Возвращаемый результат |
|-----|----------------------|-------------|-----------------|-----------|---------------------|----------------------|
| 4.1 | Backend определяет следующий шаг | - | Внутренняя логика: проверка order_num, последняя тема модуля? | - | - | next_theme_id / module_completed |
| 4.2 | Для пользователей без подписки: разблокировка следующей темы | - | Логика в submit-test | - | `user_progress` (создание записи для next_theme с возможностью доступа) | - |
| 4.3 | Бот показывает результат и следующий шаг | Сообщение + кнопка "Перейти к теме X" или "Начать модуль Y" | - | - | - | - |

### 4.5 Альтернативные сценарии

#### 4.5.1 Нет доступа к теме (подписка требуется)

| Условие | Действие системы | Возвращаемое сообщение |
|---------|------------------|----------------------|
| Попытка доступа к заблокированной теме (для пользователей БЕЗ подписки) | HTTP 403 Forbidden | "Доступ к этой теме ограничен. Пройдите предыдущую тему: '{required_theme_name}' или оформите подписку." |

#### 4.5.2 Тест не пройден (score < passing_score)

| Условие | Действие системы | Возвращаемое сообщение |
|---------|------------------|----------------------|
| score < passing_score | status=failed, can_retry=true | "Тест не пройден. Ваш балл: {score}%. Проходной балл: {passing_score}%. [Повторить материал] [Пересдать тест]" |

#### 4.5.3 Подписка истекла во время теста

| Условие | Действие системы | Возвращаемое сообщение |
|---------|------------------|----------------------|
| Проверка подписки в submit-test: expires_at < NOW() | HTTP 403 Forbidden | "Ваша подписка истекла. Продлите подписку для доступа к тестам." |

### 4.6 Зависимости

#### Backend Endpoints
**Используемые API endpoints**:
- `GET /api/v1/content/modules` - получение списка модулей
- `GET /api/v1/content/modules/{id}/themes` - получение списка тем модуля
- `POST /api/v1/users/{user_id}/study-sessions` - создание сессии изучения темы
- `POST /api/v1/users/{user_id}/test-attempts` - начало прохождения теста
- `PUT /api/v1/users/{user_id}/test-attempts/{attempt_id}` - завершение теста (отправка ответов)

#### База данных
- `modules`: SELECT (id, name, description, order_num, is_locked)
- `themes`: SELECT, JOIN с user_progress
- `mnemonics`: SELECT (theme_id, type, content_text, s3_image_key, order_num)
- `tests`: SELECT (theme_id, questions_json, difficulty, passing_score)
- `user_progress`: SELECT, INSERT, UPDATE (user_id, theme_id, status, score, attempts, current_attempt, test_started_at, started_at, completed_at)
- `test_attempts`: INSERT (новая таблица для истории)
- `subscriptions`: SELECT (для проверки доступа)

**Примечание**: Поля `current_attempt` и `test_started_at` в таблице `user_progress` используются только для статистики и аналитики. Система НЕ ограничивает количество попыток прохождения тестов или время на их выполнение.

#### Внешние сервисы
- **S3 Storage**: Генерация pre-signed URLs для изображений мнемоник
- **Telegram Bot API**: Отправка изображений, inline keyboard

#### Новые поля БД (требуются)
- `user_progress.current_attempt` (INT, default 0) - для статистики
- `user_progress.test_started_at` (TIMESTAMP) - для статистики
- `user_progress.started_at` (TIMESTAMP)
- `themes.is_locked` (BOOLEAN, default false)
- `modules.is_locked` (BOOLEAN, default false)

#### Новая таблица `test_attempts`
```sql
CREATE TABLE test_attempts (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    theme_id INT NOT NULL,
    test_id INT NOT NULL,
    answers JSONB NOT NULL,
    score INT NOT NULL,
    passed BOOLEAN NOT NULL,
    started_at TIMESTAMP NOT NULL,
    submitted_at TIMESTAMP NOT NULL,
    duration_seconds INT NOT NULL
);
```

## 5. User Flow 3: Активация промокода преподавателем

### 5.1 Описание

Преподаватель получает промокод от администратора и активирует его для своих студентов.

### 5.2 Участники

- **Актор**: Преподаватель
- **Системы**: Telegram Bot, Backend Service, SQLite Database

### 5.3 Предусловия

- Пользователь зарегистрирован с ролью "teacher"
- Администратор создал промокод через GoAdmin
- Промокод в статусе "pending" (не активирован)

### 5.4 Основной сценарий

| Шаг | Действие пользователя | UI компонент | Backend endpoint | Параметры | Изменяемые сущности | Возвращаемый результат |
|-----|----------------------|-------------|-----------------|-----------|---------------------|----------------------|
| 1 | Преподаватель вводит промокод | `/activate_code ABC123` | `POST /api/v1/teachers/{teacher_id}/promo-codes` | `code` (в body) | `promo_codes` (UPDATE: teacher_id, activated_at=NOW(), status=active) | `{code, university_name, max_activations, remaining, expires_at, status, activated_at}` (201 Created) |
| 2 | Backend проверяет промокод | - | SELECT FROM promo_codes WHERE code=? AND status='pending' | - | - | promo_data |
| 3 | Backend проверяет роль | - | SELECT FROM users WHERE telegram_id=? AND role='teacher' | - | - | - |
| 4 | Backend активирует промокод | - | UPDATE promo_codes SET teacher_id=?, activated_at=NOW(), status='active' | - | `promo_codes` | - |
| 5 | Бот показывает информацию о промокоде | Сообщение + инструкция для студентов | - | - | - | "Промокод активирован! Передайте студентам: /join_university ABC123" |

### 5.5 Альтернативные сценарии

## 6. User Flow 4: Присоединение студента по промокоду

### 6.1 Описание

### 6.2 Участники

- **Актор**: Студент
- **Системы**: Telegram Bot, Backend Service, SQLite Database

### 6.3 Предусловия

- Студент зарегистрирован
- Преподаватель активировал промокод
- Промокод в статусе "active"
- remaining > 0 (есть доступные активации)

### 6.4 Основной сценарий

| Шаг | Действие пользователя | UI компонент | Backend endpoint | Параметры | Изменяемые сущности | Возвращаемый результат |

| 5 | Backend создаёт/обновляет подписку | - | INSERT INTO subscriptions (user_id, type=university, status=active, plan=promo_code) | - | `subscriptions` | - |

| 7 | Backend уменьшает счётчик промокода | - | UPDATE promo_codes SET remaining=remaining-1 WHERE code=? | - | `promo_codes` | - |

## 7. User Flow 5: Оформление персональной подписки

### 7.1 Описание

Студент оформляет платную подписку через встроенную систему оплаты для получения полного доступа к контенту.

### 7.2 Участники

- **Актор**: Студент
- **Системы**: Telegram Bot, Backend Service, SQLite Database, Payment Gateway (Stripe или YooKassa)

### 7.3 Предусловия

- Пользователь зарегистрирован
- Нет активной подписки (или готов оплатить новую)
- Платёжный шлюз настроен и доступен

### 7.4 Основной сценарий

#### Этап 1: Создание счёта на оплату

| Шаг | Действие пользователя | UI компонент | Backend endpoint | Параметры | Изменяемые сущности | Возвращаемый результат |
|-----|----------------------|-------------|-----------------|-----------|---------------------|----------------------|
| 1.1 | Студент запрашивает подписку | `/subscribe` | `POST /api/v1/subscriptions/create-invoice` | `user_id`, `plan` (monthly/yearly) | `users` (UPDATE: pending_payment_id) | `{payment_id, payment_url, amount, currency, plan, expires_at}` |
| 1.2 | Backend проверяет активную подписку | - | SELECT FROM subscriptions WHERE user_id=? AND status='active' AND expires_at > NOW() | - | - | - |
| 1.3 | Backend создаёт счёт в Payment Gateway | - | Payment Gateway API: CreateInvoice(amount, user_id, plan) | - | - | payment_id, payment_url |
| 1.4 | Backend сохраняет pending payment | - | UPDATE users SET pending_payment_id=? WHERE telegram_id=? | - | `users` | - |
| 1.5 | Бот показывает тарифы и ссылку на оплату | Inline keyboard: "Оплатить {amount} ₽" (URL кнопка) + "Проверить статус оплаты" | - | - | - | - |

#### Этап 2: Обработка оплаты (webhook)

| Шаг | Действие системы | Триггер | Backend endpoint | Параметры | Изменяемые сущности | Возвращаемый результат |
|-----|------------------|---------|-----------------|-----------|---------------------|----------------------|
| 2.1 | Payment Gateway отправляет webhook | Успешная оплата | `POST /api/v1/webhooks/payment-callback` | `payment_id`, `status`, `user_id`, `amount`, `signature` | `subscriptions` (INSERT), `users` (UPDATE) | HTTP 200 OK |
| 2.2 | Backend верифицирует подпись | - | HMAC SHA256 verification | - | - | valid / invalid |
| 2.3 | Backend проверяет идемпотентность | - | SELECT FROM subscriptions WHERE payment_id=? | - | - | EXISTS / NOT EXISTS |
| 2.4 | Backend создаёт подписку (если status=success) | - | INSERT INTO subscriptions (user_id, payment_id, type=personal, status=active, plan, amount, expires_at=NOW()+30days) | - | `subscriptions` | - |
| 2.5 | Backend обновляет пользователя | - | UPDATE users SET subscription_status='active', pending_payment_id=NULL WHERE telegram_id=? | - | `users` | - |
| 2.6 | Backend отправляет уведомление | - | Telegram API: send_message(user_id, "Подписка активирована!") | - | - | - |

#### Этап 3: Проверка статуса оплаты (опционально)

| Шаг | Действие пользователя | UI компонент | Backend endpoint | Параметры | Изменяемые сущности | Возвращаемый результат |
|-----|----------------------|-------------|-----------------|-----------|---------------------|----------------------|
| 3.1 | Студент проверяет статус | `/check_payment` или Callback: `check_payment_status` | `GET /api/v1/subscriptions/{user_id}/check-payment` | `user_id` | - | `{has_pending_payment: true/false, payment_id, status: pending/success/failed, created_at}` |
| 3.2 | Backend проверяет pending_payment_id | - | SELECT users WHERE telegram_id=? | - | - | - |
| 3.3 | Backend запрашивает статус у Payment Gateway | - | Payment Gateway API: GetPaymentStatus(pending_payment_id) | - | - | status |
| 3.4 | Бот показывает статус | Сообщение | - | - | - | "Оплата получена!" / "Ожидание оплаты" / "Оплата отклонена" |

### 7.5 Альтернативные сценарии

#### 7.5.1 Активная подписка уже существует

| Условие | Действие системы | Возвращаемое сообщение |
|---------|------------------|----------------------|
| EXISTS active subscription | HTTP 409 Conflict | "У вас уже есть активная подписка (до {expires_at}). Для изменения тарифа обратитесь в поддержку." |

#### 7.5.2 Платёж отклонён (webhook status=failed)

| Условие | Действие системы | Возвращаемое сообщение |
|---------|------------------|----------------------|
| webhook: status='failed' | UPDATE users SET pending_payment_id=NULL, отправка уведомления | "Оплата отклонена платёжной системой. Попробуйте другую карту или способ оплаты." |

#### 7.5.3 Дублирование webhook

| Условие | Действие системы | Возвращаемое сообщение |
|---------|------------------|----------------------|
| payment_id уже существует в subscriptions | HTTP 200 OK (идемпотентность), пропустить INSERT | (нет сообщения пользователю) |

#### 7.5.4 Невалидная подпись webhook

| Условие | Действие системы | Возвращаемое сообщение |
|---------|------------------|----------------------|
| Signature verification failed | HTTP 400 Bad Request, логирование security alert | (нет обработки на стороне пользователя) |

#### 7.5.5 Пользователь закрыл страницу оплаты

| Условие | Действие системы | Возвращаемое сообщение |
|---------|------------------|----------------------|
| pending_payment_id висит > 24 часов | Cronjob: UPDATE users SET pending_payment_id=NULL (очистка) | (нет сообщения, можно создать новый счёт) |

### 7.6 Зависимости

#### Backend Endpoints
- `POST /api/v1/subscriptions/create-invoice` ❌ новый (критический)
- `POST /api/v1/webhooks/payment-callback` ❌ новый (критический)
- `GET /api/v1/subscriptions/{user_id}/check-payment` ❌ новый (средний приоритет)

#### База данных
- `users`: SELECT, UPDATE (pending_payment_id, subscription_status)
- `subscriptions`: SELECT (проверка активной), INSERT

#### Новые поля БД (требуются)
- `users.pending_payment_id` (VARCHAR(255)) - критический

#### Внешние сервисы
- **Payment Gateway (Stripe или YooKassa)**:
  - CreateInvoice API - создание счёта
  - GetPaymentStatus API - проверка статуса
  - Webhook - уведомление об оплате
- **Telegram Bot API**: Отправка уведомлений

## 10. User Flow 8: Переход между темами и модулями

### 10.2 Участники

### 10.3 Предусловия

- Студент завершил тест с проходным баллом (score >= passing_score)
- Существует следующая тема или модуль

### 10.4 Основной сценарий

| Шаг | Действие системы | Триггер | Backend endpoint | Параметры | Изменяемые сущности | Возвращаемый результат |
|-----|------------------|---------|-----------------|-----------|---------------------|----------------------|
| 1 | Backend определяет следующий шаг | Успешная отправка теста | Внутренняя логика в `POST /user/submit-test` | - | - | next_theme_id / module_completed |
| 2 | Backend проверяет: последняя тема модуля? | - | SELECT themes WHERE module_id=? AND order_num > current_theme.order_num ORDER BY order_num LIMIT 1 | - | - | next_theme / NULL |
| 3a | Если не последняя: разблокировка следующей темы | - | UPDATE или логика для пользователей без подписки | - | - | next_theme_id |
| 3b | Если последняя: открытие следующего модуля | - | SELECT modules WHERE order_num > current_module.order_num ORDER BY order_num LIMIT 1 | - | - | next_module_id |
| 4 | Для пользователей без подписки: создание доступа | - | INSERT INTO user_progress (user_id, theme_id=next_theme, status=accessible) или использование is_locked=false | - | `user_progress` (возможно) | - |
| 5 | Backend формирует ответ | - | - | - | - | `{next_action: {type: "next_theme" / "next_module" / "course_completed", theme_id, module_id, message}}` |
| 6 | Бот показывает кнопку перехода | Inline keyboard: "Перейти к теме X →" или "Начать модуль Y →" | - | - | - | - |
| 7 | Студент нажимает кнопку | Callback: `next_theme_{theme_id}` | `POST /user/study-theme` (переход к User Flow 2) | - | - | - |

### 13.3 Интеграция с Payment Gateway (Stripe или YooKassa)

#### Использование

**User Flow**: 5 (Оформление подписки)

**Функциональность**:
- Создание счёта на оплату (invoice)
- Получение payment URL для пользователя
- Webhook для уведомлений об оплате
- Проверка статуса платежа

#### Выбор провайдера

| Провайдер | Преимущества | Недостатки |
|-----------|--------------|------------|
| **YooKassa** | Российская система, поддержка рублей, множество способов оплаты | Комиссия ~2-3% |
| **Stripe** | Международная система, отличная документация | Требует валютные счета, сложности с РФ |

**Рекомендация**: YooKassa для российской аудитории, Stripe для международной.

#### Требования (YooKassa)

**Создание платежа**:
```go
// POST https://api.yookassa.ru/v3/payments
{
    "amount": {
        "value": "500.00",
        "currency": "RUB"
    },
    "confirmation": {
        "type": "redirect",
        "return_url": "https://t.me/your_bot"
    },
    "capture": true,
    "description": "Подписка на месяц",
    "metadata": {
        "user_id": "123456789",
        "plan": "monthly"
    }
}
// Response: {id: "payment_id", confirmation: {confirmation_url}}
```

**Webhook**:
- URL: `POST /api/v1/webhooks/payment-callback`
- Верификация: IP whitelist YooKassa или HMAC signature
- Поля: `event: "payment.succeeded"`, `object: {id, status, paid, amount, metadata}`
- Идемпотентность: проверка payment_id перед обработкой

**Проверка статуса**:
```go
// GET https://api.yookassa.ru/v3/payments/{payment_id}
// Response: {id, status, paid, amount}
```

#### Требования безопасности
- ✅ Верификация webhook подписи (HMAC SHA256)
- ✅ IP whitelist для webhook
- ✅ Идемпотентность обработки webhook
- ✅ Логирование всех платежей
- ✅ Хранение payment_id в БД

## 14. Обработка ошибок

### 14.1 Классификация ошибок

| Категория | HTTP код | Примеры | Действие системы |
|-----------|----------|---------|------------------|
| **Валидация** | 400 Bad Request | Невалидные параметры, неполные ответы теста | Показать сообщение с описанием ошибки |
| **Авторизация** | 401 Unauthorized | Неверный telegram_id | Попросить перезапустить бота |
| **Доступ** | 403 Forbidden | Нет доступа к теме, превышен лимит попыток | Показать причину и предложение (подписка/пересдача) |
| **Не найдено** | 404 Not Found | Тема/модуль не существует, промокод не найден | Показать сообщение "не найдено" |
| **Конфликт** | 409 Conflict | Промокод уже активирован, дубликат платежа | Показать информативное сообщение |
| **Сервер** | 500 Internal Server Error | Ошибка БД, недоступен S3 | Показать "Попробуйте позже", залогировать |

### 14.2 Сообщения об ошибках для пользователя

#### Примеры хороших сообщений

✅ **Хорошо** (информативно + действие):
```
⚠️ Доступ к этой теме ограничен.

Вам нужно:
1. Пройти предыдущую тему: "Позвоночник"
   [Перейти к теме]

ИЛИ

2. Оформить подписку для полного доступа
   [Оформить подписку]
```

❌ **Плохо** (непонятно):
```
Ошибка доступа
```

### 14.3 Логирование

#### Что логировать

| Событие | Уровень | Данные |
|---------|---------|--------|
| Регистрация пользователя | INFO | telegram_id, role, timestamp |
| Прохождение теста | INFO | user_id, theme_id, score, passed, timestamp |
| Активация промокода | INFO | teacher_id, code, timestamp |
| Оплата подписки (webhook) | INFO | payment_id, user_id, amount, status, timestamp |
| Ошибка доступа (403) | WARN | user_id, resource, reason, timestamp |
| Ошибка валидации (400) | WARN | user_id, endpoint, validation_error, timestamp |
| Ошибка БД | ERROR | query, error_message, stack_trace, timestamp |
| Невалидная webhook подпись | ERROR | payment_id, signature, ip_address, timestamp |
| Panic в backend | CRITICAL | error_message, stack_trace, timestamp |

#### Формат (structured logging)
```json
{
  "timestamp": "2025-12-28T10:30:00Z",
  "level": "INFO",
  "event": "test_completed",
  "user_id": 123456789,
  "theme_id": 2,
  "score": 85,
  "passed": true,
  "attempt": 1
}
```
